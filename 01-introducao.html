1  Introdução a Bancos de Dados

Nota

Este capítulo é baseado no Capítulo 1 de Silberschatz, Korth, e Sudarshan (2019).

1.1 O que é um Banco de Dados?

Um banco de dados é uma coleção organizada de dados inter-relacionados que modela algum aspecto do mundo real. Um Sistema de Gerenciamento de Banco de Dados (SGBD) é o software que permite criar, manter e consultar esses dados de forma eficiente, segura e confiável.

Exemplos do cotidiano que dependem de bancos de dados:

  • Sistemas bancários (contas, transações, empréstimos)
  • Redes sociais (usuários, postagens, conexões)
  • E-commerce (produtos, pedidos, clientes)
  • Sistemas hospitalares (prontuários, agendamentos)
  • Aplicativos de navegação (mapas, rotas, pontos de interesse)

1.2 Aplicações de Banco de Dados

Bancos de dados estão presentes em praticamente todos os setores:

Empresas e Negócios

  • Vendas: clientes, produtos, compras
  • Contabilidade: pagamentos, recebimentos, ativos
  • RH: funcionários, salários, impostos

Indústria

  • Gestão de produção, inventário, pedidos e cadeia de suprimentos

Setor Financeiro e Bancário

  • Informações de clientes, contas e empréstimos
  • Transações com cartão de crédito
  • Negociação de ações e títulos em tempo real

Universidades

  • Matrículas, notas, histórico escolar

Telecomunicações

  • Registros de chamadas, SMS e uso de dados
  • Geração de faturas mensais

Serviços Web

  • Varejistas online: rastreamento de pedidos, recomendações personalizadas
  • Publicidade online direcionada

Sistemas de Navegação

  • Localização de pontos de interesse, rotas de estradas, linhas de transporte

1.3 Por que não usar Arquivos?

Nos primórdios da computação, sistemas de informação eram construídos diretamente sobre sistemas de arquivos. Essa abordagem apresenta problemas sérios:

1.3.1 Problemas com Sistemas de Arquivos

Problema Descrição
Redundância e inconsistência Os mesmos dados são armazenados em múltiplos arquivos, em formatos diferentes, levando à duplicação e inconsistências
Dificuldade de acesso É preciso escrever um novo programa para cada nova necessidade de consulta
Isolamento de dados Dados em múltiplos arquivos e formatos dificultam consultas integradas
Problemas de integridade Restrições (ex: saldo > 0) ficam enterradas no código dos programas, difíceis de modificar
Atomicidade de atualizações Falhas podem deixar o sistema em estado inconsistente (ex: transferência bancária parcial)
Acesso concorrente Acessos simultâneos sem controle levam a resultados incorretos
Problemas de segurança Difícil controlar o acesso granular aos dados
Importante

Sistemas de banco de dados oferecem soluções para todos esses problemas.

1.4 Modelos de Dados

Um modelo de dados é um conjunto de ferramentas conceituais para descrever:

  • Os dados em si
  • As relações entre os dados
  • A semântica (significado) dos dados
  • As restrições (constraints) sobre os dados

1.4.1 Principais Modelos

Modelos de Dados
├── Modelo Relacional          ← foco principal desta disciplina
├── Modelo Entidade-Relacionamento (ER)  ← para projeto de banco de dados
├── Modelos Baseados em Objetos
│   ├── Orientado a Objetos
│   └── Objeto-Relacional
├── Modelo Semiestruturado (XML, JSON)
└── Modelos Antigos (legados)
    ├── Modelo em Rede
    └── Modelo Hierárquico

1.4.2 Modelo Relacional

No modelo relacional (proposto por Ted Codd em 1970, Prêmio Turing 1981), todos os dados são armazenados em tabelas (relações):

  • Cada tabela possui colunas (atributos)
  • Cada linha é uma tupla (registro)

Exemplo de tabela de instrutores:

ID nome depto salário
10101 Srinivasan Comp. Sci. 65.000,00
12121 Wu Finance 90.000,00
15151 Mozart Music 40.000,00
22222 Einstein Physics 95.000,00

1.5 Visão em Três Níveis (Arquitetura ANSI/SPARC)

Um SGBD organiza os dados em três níveis de abstração:

┌──────────────────────────────────────┐
│  Nível de Visão (View Level)         │  ← o que cada usuário vê
│  Visão 1  |  Visão 2  |  Visão 3    │
├──────────────────────────────────────┤
│  Nível Lógico (Logical Level)        │  ← estrutura lógica completa
│  (quais dados existem e suas         │
│   relações)                          │
├──────────────────────────────────────┤
│  Nível Físico (Physical Level)       │  ← como os dados são armazenados
│  (estruturas de armazenamento)       │
└──────────────────────────────────────┘

Essa separação é fundamental para a independência de dados.

1.5.1 Schemas e Instâncias

Analogia com linguagens de programação:

Banco de Dados Programação
Schema Lógico Tipo de uma variável
Schema Físico Layout de memória
Instância Valor atual da variável
  • Schema lógico: estrutura geral do banco (ex: “existe uma tabela de clientes com os campos nome, CPF, endereço”)
  • Schema físico: como os dados são fisicamente armazenados em disco
  • Instância: o conteúdo atual do banco em um dado momento

1.5.2 Independência de Dados Física

A independência de dados física é a capacidade de modificar o schema físico sem alterar o schema lógico. Isso permite, por exemplo, reorganizar como os dados são guardados em disco sem impactar os programas que os utilizam.

1.6 Linguagens de Banco de Dados

1.6.1 DDL — Data Definition Language

A DDL é usada para definir a estrutura do banco de dados (schema).

CREATE TABLE instructor (
    ID          CHAR(5),
    name        VARCHAR(20)   NOT NULL,
    dept_name   VARCHAR(20),
    salary      NUMERIC(8,2),
    PRIMARY KEY (ID)
);

O compilador DDL gera um conjunto de templates de tabelas armazenados no dicionário de dados, que contém metadados como:

  • Schema do banco
  • Restrições de integridade (chave primária, chave estrangeira)
  • Autorizações de acesso

1.6.2 DML — Data Manipulation Language

A DML (também chamada de linguagem de consulta) é usada para acessar e atualizar dados. Existem dois tipos:

Tipo Característica Exemplo
Procedural Especifica o que e como obter os dados Álgebra Relacional
Declarativa Especifica apenas o que obter SQL

DMLs declarativas são mais fáceis de aprender e usar. SQL é o padrão comercial mais utilizado.

1.6.3 SQL — Structured Query Language

SQL é não-procedural: uma consulta recebe uma ou mais tabelas e sempre retorna uma tabela.

-- Encontrar todos os instrutores do departamento de Ciência da Computação
SELECT name
FROM instructor
WHERE dept_name = 'Comp. Sci.';
Nota

SQL não é equivalente a uma Máquina de Turing. Para computações complexas, SQL é geralmente embutido em uma linguagem de propósito geral (Python, Java, C++).

Formas de acessar o banco a partir de aplicações:

  1. Extensões de linguagem para SQL embutido
  2. Interfaces de programa (ODBC, JDBC, drivers de banco de dados)

1.7 Projeto de Banco de Dados

O projeto de um banco de dados envolve duas etapas principais:

Projeto Lógico
└── Decidir o schema: quais tabelas existirão e quais atributos cada uma terá
    ├── Decisão de negócio: quais dados precisamos registrar?
    └── Decisão técnica: como distribuir atributos entre as tabelas?

Projeto Físico
└── Decidir o layout físico: índices, particionamento, armazenamento

1.8 Motor de Banco de Dados (Database Engine)

Um SGBD é dividido em módulos que gerenciam diferentes responsabilidades:

1.8.1 Gerenciador de Armazenamento (Storage Manager)

Provê a interface entre os dados de baixo nível armazenados e os programas e consultas. É responsável por:

  • Interação com o gerenciador de arquivos do SO
  • Armazenamento, recuperação e atualização eficiente dos dados

Componentes:

  • Gerenciador de autorização e integridade: verifica restrições
  • Gerenciador de transações: garante consistência em falhas
  • Gerenciador de arquivos: gerencia a alocação de espaço em disco
  • Gerenciador de buffer: carrega dados do disco para a memória RAM

Estruturas de dados:

  • Arquivos de dados: armazenam o banco em si
  • Dicionário de dados: armazena metadados e o schema
  • Índices: fornecem acesso rápido a itens de dados por valor

1.8.2 Processador de Consultas (Query Processor)

Componentes:

  • Interpretador DDL: interpreta instruções DDL e registra definições no dicionário de dados
  • Compilador DML: traduz DML em um plano de execução de instruções de baixo nível (com query optimization)
  • Motor de avaliação de consultas: executa as instruções geradas pelo compilador

Etapas do processamento de uma consulta:

1.8.3 Gerenciamento de Transações

Uma transação é uma coleção de operações que executa uma única função lógica. Por exemplo, uma transferência bancária (débito de uma conta + crédito em outra) deve ser atômica.

O gerenciador de transações garante que o banco permaneça em estado consistente mesmo diante de:

  • Falhas de sistema (quedas de energia, crashes)
  • Falhas de transação
  • Acessos concorrentes

O gerenciador de controle de concorrência coordena transações simultâneas para garantir consistência.

1.9 Arquiteturas de Banco de Dados

1.9.1 Arquitetura Centralizada

  • Um servidor (ou poucos núcleos com memória compartilhada)
  • Toda a carga de trabalho é processada em um único local

1.9.2 Arquitetura Cliente-Servidor

  • Um servidor executa o trabalho em nome de múltiplos clientes
  • O cliente envia requisições SQL; o servidor processa e retorna resultados

1.9.3 Banco de Dados Paralelo

  • Memória compartilhada: múltiplos núcleos acessam a mesma RAM
  • Disco compartilhado: múltiplos processadores acessam os mesmos discos
  • Sem compartilhamento (Shared-nothing): cada nó tem sua própria memória e disco

1.9.4 Banco de Dados Distribuído

  • Dados distribuídos geograficamente
  • Heterogeneidade de schemas e dados

1.9.5 Aplicações de Duas e Três Camadas

┌─────────────────────────────────────────────────────────┐
│  Duas Camadas (Two-tier)                                 │
│                                                          │
│  [Cliente]  ←──── chamadas diretas ao BD ────→  [BD]    │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  Três Camadas (Three-tier)                               │
│                                                          │
│  [Cliente] ←→ [Servidor de Aplicação] ←→ [BD]           │
│  (navegador)  (lógica de negócio)     (dados)           │
└─────────────────────────────────────────────────────────┘

Na arquitetura de três camadas, o cliente não faz chamadas diretas ao banco — ele se comunica com um servidor de aplicação que, por sua vez, acessa o banco. Essa é a arquitetura predominante em aplicações web modernas.

1.10 Usuários de Banco de Dados

Existem quatro perfis de usuários:

Perfil Descrição
Usuários leigos Interagem com o sistema por meio de aplicações prontas (ex: caixa eletrônico, e-commerce)
Programadores de aplicação Profissionais que escrevem os programas que os usuários leigos utilizam
Usuários sofisticados Interagem com o sistema via linguagem de consulta ou ferramentas de análise de dados, sem escrever programas
Usuários especializados Desenvolvem aplicações especializadas (ex: CAD, dados geográficos, multimídia)

1.10.1 Administrador de Banco de Dados (DBA)

O DBA (Database Administrator) tem controle central sobre o sistema. Suas funções incluem:

  • Definição do schema
  • Definição de estruturas de armazenamento e métodos de acesso
  • Modificação do schema e da organização física
  • Concessão de autorizações de acesso
  • Manutenção de rotina (backups, monitoramento, espaço em disco)

1.11 Breve História dos Bancos de Dados

Período Marco
Anos 1950 e início dos 60 Processamento de dados em fitas magnéticas (acesso sequencial); cartões perfurados para entrada
Final dos anos 60 e 70 Discos rígidos permitem acesso direto; modelos em rede e hierárquico; Ted Codd define o modelo relacional; Oracle lança o primeiro BD relacional comercial
Anos 80 SQL se torna padrão industrial; BDs relacionais comerciais; BDs paralelos e distribuídos; BDs orientados a objetos
Anos 90 Data warehouses multi-terabyte; suporte a decisão e mineração de dados; surgimento do e-commerce
Anos 2000 Big data (Google BigTable, Amazon); sistemas NoSQL; MapReduce
Anos 2010 SQL sobre MapReduce; BDs massivamente paralelos; BDs em memória principal (in-memory)
