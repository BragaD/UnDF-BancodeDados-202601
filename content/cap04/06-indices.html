Índices

Autor

Douglas Braga

Muitas consultas acessam apenas uma pequena fração das tuplas de uma tabela. Percorrer todas as linhas para encontrar as que satisfazem um predicado — varredura sequencial (sequential scan) — é ineficiente quando a tabela é grande. Um índice resolve esse problema.

O Que é um Índice

Um índice sobre um atributo de uma relação é uma estrutura de dados que permite ao SGBD encontrar as tuplas que possuem um valor específico para aquele atributo sem precisar percorrer todas as tuplas da relação.

A sintaxe para criar um índice é:

CREATE INDEX <nome> ON <relação> (<atributo>);

O índice mais comum em SGBDs relacionais é a B-tree (árvore-B), que suporta eficientemente buscas por igualdade, comparações e ordenações. O PostgreSQL também oferece índices HASH, GIN e GiST para casos específicos.

Demonstração com Dados Reais

Nota

Os exemplos desta seção utilizam a base de dados completa da UnDF — com 2.720 alunos, 730 disciplinas, 6.172 ofertas de turmas e 71.053 matrículas — em vez da amostra com 18 alunos usada nos capítulos anteriores. Com esse volume de dados, o impacto dos índices é perceptível nos tempos de execução.

SELECT
  'aluno'                AS tabela, COUNT(*) AS linhas FROM aluno
UNION ALL
SELECT 'disciplina',               COUNT(*)           FROM disciplina
UNION ALL
SELECT 'ministra',                 COUNT(*)           FROM ministra
UNION ALL
SELECT 'matricula_disciplina',     COUNT(*)           FROM matricula_disciplina
ORDER BY linhas DESC;

tabela linhas
matricula_disciplina 71053
ministra 6172
aluno 2720

Sem índice — varredura sequencial

Vamos buscar todas as matrículas de um aluno específico. Primeiro, sem nenhum índice extra sobre id_aluno:

EXPLAIN ANALYZE
SELECT id_disciplina, nota, aprovado
FROM   matricula_disciplina
WHERE  id_aluno = 2023111001;
5 records
QUERY PLAN
Seq Scan on matricula_disciplina (cost=0.00..1519.20 rows=377 width=18) (actual time=0.007..4.173 rows=5 loops=1)
Filter: (id_aluno = 2023111001)
Rows Removed by Filter: 71048
Planning Time: 0.042 ms
Execution Time: 4.183 ms

O plano exibe Seq Scan com rows=71053: o PostgreSQL lê a tabela inteira para encontrar as matrículas do aluno. O campo actual time mostra o tempo real gasto.

Com índice — acesso direto

Criamos um índice B-tree sobre id_aluno:

CREATE INDEX idx_mat_aluno ON matricula_disciplina (id_aluno);

A mesma consulta agora usa o índice:

EXPLAIN ANALYZE
SELECT id_disciplina, nota, aprovado
FROM   matricula_disciplina
WHERE  id_aluno = 2023111001;
7 records
QUERY PLAN
Bitmap Heap Scan on matricula_disciplina (cost=7.04..538.74 rows=355 width=18) (actual time=0.022..0.023 rows=5 loops=1)
Recheck Cond: (id_aluno = 2023111001)
Heap Blocks: exact=1
-> Bitmap Index Scan on idx_mat_aluno (cost=0.00..6.96 rows=355 width=0) (actual time=0.019..0.019 rows=5 loops=1)
Index Cond: (id_aluno = 2023111001)
Planning Time: 0.106 ms
Execution Time: 0.033 ms

O plano muda para Index Scan (ou Bitmap Index Scan). O PostgreSQL percorre a árvore B-tree em O(log n) para localizar as entradas de 2023111001, depois acessa diretamente as páginas de disco onde essas linhas estão — sem ler o restante da tabela.

Criando Índices

-- Índice simples
CREATE INDEX idx_aluno_curso ON aluno (id_curso);

-- Índice composto: acelera filtros que combinam ano E semestre
CREATE INDEX idx_ministra_ano_sem ON ministra (ano, semestre);

-- Índice único: garante unicidade além de acelerar buscas
CREATE UNIQUE INDEX idx_prof_nome_escola ON professor (nome, id_escola);

Nota

Um índice composto (a, b) também serve para buscas só por a (o prefixo), mas não para buscas só por b. Se precisar filtrar apenas pelo segundo componente, crie um índice separado.

Por Que Índices Aceleram Consultas

Sem índice, para executar:

SELECT * FROM aluno WHERE id_curso = 311;

o SGBD lê cada linha da tabela — O(n) operações de I/O.

Com um índice B-tree sobre id_curso, o SGBD percorre a árvore em O(log n) para encontrar as entradas com id_curso = 311, depois acessa diretamente as páginas de disco onde essas linhas estão. Para uma tabela de 1 milhão de linhas, isso reduz de ~1.000.000 leituras para ~20.

Cardinalidade: Quando o Índice Não Ajuda

Cardinalidade é o número de valores distintos de uma coluna. Colunas com baixa cardinalidade (como aprovado, que assume apenas 0 ou 1) são más candidatas a índice: quando metade das linhas satisfaz o predicado, o SGBD pode preferir varredura sequencial — percorrer o índice para então acessar metade da tabela pode ser mais lento do que ler tudo sequencialmente, ou o ganho ser marginal.

EXPLAIN ANALYZE
SELECT COUNT(*) FROM matricula_disciplina WHERE aprovado = 1;
6 records
QUERY PLAN
Aggregate (cost=1465.05..1465.06 rows=1 width=8) (actual time=8.877..8.878 rows=1 loops=1)
-> Seq Scan on matricula_disciplina (cost=0.00..1464.16 rows=355 width=0) (actual time=0.013..6.375 rows=65328 loops=1)
Filter: (aprovado = 1)
Rows Removed by Filter: 5725
Planning Time: 0.038 ms
Execution Time: 8.888 ms 
CREATE INDEX idx_mat_aprovado ON matricula_disciplina (aprovado);
EXPLAIN ANALYZE
SELECT COUNT(*) FROM matricula_disciplina WHERE aprovado = 1;
8 records
QUERY PLAN
Aggregate (cost=539.63..539.64 rows=1 width=8) (actual time=8.672..8.672 rows=1 loops=1)
-> Bitmap Heap Scan on matricula_disciplina (cost=7.04..538.74 rows=355 width=0) (actual time=1.447..6.144 rows=65328 loops=1)
Recheck Cond: (aprovado = 1)
Heap Blocks: exact=523
-> Bitmap Index Scan on idx_mat_aprovado (cost=0.00..6.96 rows=355 width=0) (actual time=1.389..1.389 rows=65328 loops=1)
Index Cond: (aprovado = 1)
Planning Time: 0.117 ms
Execution Time: 8.686 ms

Observe que o plano provavelmente ignora o índice e usa Seq Scan — o otimizador estimou que percorrer metade da tabela via índice custaria mais do que ler tudo em sequência.

Tradeoffs dos Índices

Aviso

Índices não são gratuitos. Cada índice:

  • Ocupa espaço em disco — pode ser significativo em tabelas grandes.
  • Retarda inserções, atualizações e exclusões — o SGBD precisa manter o índice sincronizado com os dados.
  • Não é sempre utilizado — o otimizador decide quando usar um índice com base em estatísticas da tabela.

Boas candidatas a índice: colunas usadas frequentemente em WHERE, JOIN ON, ORDER BY com alta cardinalidade (muitos valores distintos).

Más candidatas: colunas booleanas ou com poucos valores distintos (como aprovado).

Índice Parcial

O PostgreSQL permite criar índices parciais — que indexam apenas as linhas que satisfazem um predicado. São menores e mais eficientes quando o caso de uso foca em um subconjunto:

EXPLAIN ANALYZE
SELECT id_disciplina, nota
FROM   matricula_disciplina
WHERE  id_aluno = 2023111001
  AND  aprovado = 0;
9 records
QUERY PLAN
Bitmap Heap Scan on matricula_disciplina (cost=14.16..21.84 rows=2 width=16) (actual time=0.195..0.196 rows=0 loops=1)
Recheck Cond: ((aprovado = 0) AND (id_aluno = 2023111001))
-> BitmapAnd (cost=14.16..14.16 rows=2 width=0) (actual time=0.194..0.194 rows=0 loops=1)
-> Bitmap Index Scan on idx_mat_aprovado (cost=0.00..6.96 rows=355 width=0) (actual time=0.157..0.157 rows=5725 loops=1)
Index Cond: (aprovado = 0)
-> Bitmap Index Scan on idx_mat_aluno (cost=0.00..6.96 rows=355 width=0) (actual time=0.025..0.026 rows=5 loops=1)
Index Cond: (id_aluno = 2023111001)
Planning Time: 0.035 ms
Execution Time: 0.206 ms 
-- Índice apenas para matrículas reprovadas (aprovado = 0)
CREATE INDEX idx_reprovados ON matricula_disciplina (id_aluno)
WHERE aprovado = 0;
EXPLAIN ANALYZE
SELECT id_disciplina, nota
FROM   matricula_disciplina
WHERE  id_aluno = 2023111001
  AND  aprovado = 0;
6 records
QUERY PLAN
Bitmap Heap Scan on matricula_disciplina (cost=4.30..11.97 rows=2 width=16) (actual time=0.018..0.019 rows=0 loops=1)
Recheck Cond: ((id_aluno = 2023111001) AND (aprovado = 0))
-> Bitmap Index Scan on idx_reprovados (cost=0.00..4.30 rows=2 width=0) (actual time=0.018..0.018 rows=0 loops=1)
Index Cond: (id_aluno = 2023111001)
Planning Time: 0.104 ms
Execution Time: 0.033 ms

O índice parcial é menor que um índice completo sobre (id_aluno) porque armazena entradas apenas das linhas reprovadas — tipicamente uma fração da tabela.
